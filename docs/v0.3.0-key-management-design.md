# AI Router v0.3.0 Key Management 技术设计

状态：Implemented

目标版本：v0.3.0

适用范围：本地模式的真实客户端密钥管理，并为后续公网部署、多租户和 API 分发建立兼容的数据与运行边界。

## 1. 背景

AI Router v0.2.6 已支持通过 `auth.enabled` 和静态 `auth.keys` 验证客户端请求，也会在调用记录中保存 `client_key_id`。现有实现适合少量、由操作者手工维护的本地调用方，但不具备完整的凭据生命周期：

- Key 需要手工生成并通过环境变量配置；
- Key 与具体应用、权限和用量没有结构化关联；
- 无法在控制台创建、停用、过期、撤销或轮换；
- 缺少模型、协议、IP 和速率限制；
- 缺少可持久化的按 Key 请求与 Token 统计；
- 静态 YAML 不适合作为未来公网控制面的动态状态库。

v0.3.0 将新增“访问密钥”能力。该能力只管理调用 AI Router 网关的客户端凭据，不替代模型服务的上游 API Key，也不承担管理员登录认证。

## 2. 设计目标

### 2.1 v0.3.0 必须完成

1. 在控制台创建真实、不可预测的客户端 Key。
2. 支持 Key 的启用、停用、过期、撤销和无中断轮换。
3. 支持按客户端限制模型、协议、IP、请求速率和并发。
4. 支持按客户端查看请求、错误和 Token 用量。
5. 应用配置页可按名称选择托管 Key，浏览器无需读取完整 Secret。
6. 保持现有 `auth.keys` 配置兼容，并提供无中断迁移路径。
7. 本地模式不强制部署 PostgreSQL 或 Redis。
8. 数据模型和服务接口可以在后续替换为 PostgreSQL、Redis 和多租户实现。

### 2.2 非目标

v0.3.0 不包含：

- 用户注册、组织邀请和公开自助开户；
- 充值、兑换码、支付、账单和发票；
- 多实例之间的分布式限流；
- OAuth/OIDC 管理员登录和完整 RBAC；
- Kubernetes 或控制面、数据面独立部署；
- 按金额进行具有财务意义的结算。

这些能力属于 v0.4.0 及之后的公网部署路线。

## 3. 核心原则

### 3.1 三类凭据必须分离

| 凭据 | 用途 | 管理方式 |
| --- | --- | --- |
| Provider Secret | AI Router 调用上游模型 | 模型接入页面 |
| Client Credential | Codex、Claude Code、SDK 调用 AI Router | 访问密钥页面 |
| Admin Credential | 登录和操作控制面 | 现有 Admin Token，未来升级为管理员会话 |

任何 Client Credential 都不能访问管理 API，任何 Admin Credential 也不应自动获得模型调用权限。

### 3.2 Client 与 Credential 分离

`Client` 表示稳定的调用主体，例如“我的 Codex”或“CI Runner”；`Credential` 表示 Client 当前使用的一份密钥。一个 Client 可以同时拥有新旧两份 Credential，从而完成无中断轮换。

### 3.3 本地与公网采用不同的 Secret 策略

- 公网分发型 Credential：完整 Secret 只显示一次，服务端保存摘要。
- 本地托管型 Credential：可以加密恢复，用于稍后重新写入本地应用。

v0.3.0 同时实现两类 Credential：公网或导入型标准 Credential 只保存摘要；Web 控制台在本地生成的 Credential 使用 AES-256-GCM 加密保存，以便之后在应用配置页按名称重新选择。两类 Credential 都使用独立 HMAC 摘要完成网关鉴权。

### 3.4 动态状态不继续堆入 YAML

YAML 适合启动配置和人工维护的路由配置，不适合频繁变化的凭据、计数器和审计事件。v0.3.0 将 Key 状态放入独立存储，同时保留旧 `auth.keys` 作为兼容输入。

## 4. 运行架构

```text
HTTP Request
  -> Credential Extractor
  -> Credential Authenticator
  -> Client Policy Precheck
  -> Protocol Decode
  -> Model/Route Authorization
  -> Per-client Rate and Concurrency Guard
  -> Existing Router and Protocol Conversion
  -> Upstream Provider
  -> Usage Settlement
  -> Request Log and Client Usage
```

新增核心包建议如下：

```text
internal/clientauth     Credential 解析、摘要验证和 Principal
internal/clientstore    Client、Credential、Policy 持久化接口
internal/ratelimit      单机速率和并发限制
internal/usage          用量预留、结算和聚合
internal/audit          密钥管理操作审计
```

`internal/auth` 保留管理员认证，并逐步将客户端认证委托给 `internal/clientauth`。

## 5. 数据模型

### 5.1 Client

```go
type Client struct {
    ID          string
    TenantID    string
    ProjectID   string
    Name        string
    Description string
    Status      ClientStatus
    PolicyID    string
    CreatedAt   time.Time
    UpdatedAt   time.Time
}
```

v0.3.0 尚未提供租户和项目管理，但新建 Client 固定写入内部常量 `tenant_default` 与 `project_default`。这两个字段从首版数据库开始持久化，后续升级多租户时不需要重写 Client ID、Credential 关联或历史用量。

状态：

- `active`
- `disabled`
- `deleted`

删除使用软删除，历史日志和用量仍保留 Client ID 与当时名称快照。

### 5.2 Credential

```go
type Credential struct {
    ID               string
    ClientID         string
    Kind             CredentialKind
    Prefix           string
    SecretHMAC       []byte
    SecretCiphertext []byte
    SecretNonce      []byte
    HMACKeyID        string
    Status           CredentialStatus
    CreatedAt        time.Time
    ExpiresAt        *time.Time
    LastUsedAt       *time.Time
    RevokedAt        *time.Time
}
```

`Kind`：

- `standard`：不可恢复，只保存摘要；
- `managed`：加密可恢复的本地凭据，只允许后端为应用配置流程解密。

状态：

- `active`
- `disabled`
- `expired`
- `revoked`

撤销不可逆；停用可以重新启用。

### 5.3 ClientPolicy

```go
type ClientPolicy struct {
    ID                   string
    ProjectID            string
    AllowedModels        []string
    AllowedProtocols     []ir.Protocol
    AllowedCIDRs         []string
    RequestsPerMinute    int
    Burst                int
    MaxConcurrent        int
    DailyRequestLimit    int64
    DailyInputTokens     int64
    DailyOutputTokens    int64
    MaxOutputTokens      int
}
```

规则：

- 空模型集合表示允许全部当前客户端模型；
- 空协议集合表示允许全部已支持协议；
- 空 CIDR 集合表示不限制来源 IP；
- 数值为零表示不启用对应限制；
- Provider ID 和上游模型名不能作为客户端权限边界，权限以客户端请求模型和客户端协议为准；
- 不能通过请求 Header 绕过 Policy 选择其他 Route 或 Provider。

### 5.4 UsageBucket

```go
type UsageBucket struct {

    ProjectID    string
    ClientID     string

    WindowKind   string
    Requests     int64
    Errors       int64

    OutputTokens int64

```

v0.3.0 使用分钟与自然日两个窗口。请求日志仍保存单次调用详情，UsageBucket 用于快速展示和限制判断。

### 5.5 AuditEvent

```go
type AuditEvent struct {
    ID           string
    TenantID     string
    ProjectID    string
    ActorType    string
    ActorID      string
    Action       string
    ResourceType string

```

至少记录：

- 启用、停用和撤销；
- 轮换；
- 配置到应用；
- 旧配置迁移。

审计事件不得包含完整 Secret。

## 6. Key 格式与密码学

标准格式：

```text
sk-<base64url-secret>
```

要求：

1. Credential 内部 ID 使用至少 96 bit 随机标识，但不编码进完整 Key。
2. `secret` 使用 `crypto/rand` 生成 32 字节。
3. Secret 使用无填充 Base64URL 编码，并统一采用 `sk-` 前缀。
4. 存储值为 `HMAC-SHA256(masterKey, completeKey)`，而不是明文。
5. `masterKey` 从环境变量或独立 Secret 文件读取，不写入主数据库。
6. 验证使用恒定时间比较。
7. 管理 API 只在创建、导入和轮换响应中返回一次完整 Key。
8. 日志、错误、指标、追踪、URL 和审计事件必须统一脱敏。

每条 Credential 保存 `HMACKeyID`。本地首次运行自动生成 `credential-master.key`，权限为 `0600`；服务端模式要求从环境变量、容器 Secret 或 KMS 提供。验证端支持短期加载多代 HMAC Key，便于以后轮换 Master Key。Master Key 丢失会使对应 Credential 无法验证，因此备份流程必须同时覆盖该文件，但不得把它放入普通诊断包。

状态库备份必须生成带版本号的 manifest，记录数据库快照、所需 `HMACKeyID` 集合和文件校验值。恢复时先在临时目录校验数据库与全部主密钥，再原子替换运行文件；两者任一缺失或不匹配都拒绝启动托管鉴权。服务端模式只备份 Key ID 与外部 KMS 引用，不导出 KMS Secret。

Key 自带 Credential ID，因此正常验证是一次索引查询，不遍历全部 Credential。旧版静态 Key 继续走兼容验证路径。

## 7. 存储设计

### 7.1 Store 接口

```go
type Scope struct {
    TenantID  string
    ProjectID string
}

type Store interface {
    CreateClient(context.Context, Scope, Client, ClientPolicy) error
    UpdateClient(context.Context, Scope, Client) error
    GetClient(context.Context, Scope, string) (Client, error)
    ListClients(context.Context, Scope, ClientFilter) ([]ClientSummary, error)

    CreateCredential(context.Context, Scope, Credential) error
    GetCredential(context.Context, Scope, string) (Credential, error)
    UpdateCredentialStatus(context.Context, Scope, string, CredentialStatus) error
    TouchCredential(context.Context, Scope, string, time.Time) error

    GetPolicy(context.Context, Scope, string) (ClientPolicy, error)
    UpdatePolicy(context.Context, Scope, ClientPolicy) error

    AddUsage(context.Context, Scope, UsageDelta) error
    GetUsage(context.Context, Scope, UsageQuery) (UsageSummary, error)
    AppendAudit(context.Context, Scope, AuditEvent) error
}
```

所有网关和管理代码依赖接口，不直接依赖具体数据库。v0.3.0 的处理器统一传入 `DefaultScope`；Store 仍需校验记录中的范围，不能忽略参数。

### 7.2 v0.3.0 单机实现

v0.3.0 使用 `go.etcd.io/bbolt` 作为独立嵌入式状态库，路径位于 AI Router 运行目录：

```text
<runtime-dir>/gateway-state.db
```

要求：

- 运行目录为 `0700`；
- 数据库文件为 `0600`；
- 写入支持事务和崩溃恢复；
- 启动时自动执行版本化迁移；
- 支持导出不含 Secret 的诊断信息；
- 支持备份与恢复；
- 数据损坏时网关必须 fail closed，不能自动关闭认证继续运行。

选择 bbolt 的原因：

- 纯 Go，符合当前 macOS、Linux、Windows 全平台 `CGO_ENABLED=0` 构建要求；
- 单文件、事务化并支持崩溃恢复，适合现阶段单进程运行模型；
- 可以通过只读事务生成一致性备份；
- 数据库实现被 Store 接口隔离，不会阻塞后续 PostgreSQL 实现。

初始 Bucket：

```text
meta
clients
credentials
credential_by_prefix
credential_by_hmac
policies
usage_daily
usage_minute
usage_reservations
audit_events
```

所有业务 Bucket 的 Key 必须带 `ProjectID` 前缀，或其记录中包含可校验的 `ProjectID`。v0.3.0 只有默认范围也不能省略该维度，避免后续多租户迁移依赖全库推断。

所有写入通过事务执行。Credential 状态、配额预留和 Usage 增量必须在同一事务内完成，不能依赖异步日志推导权限状态。bbolt 的单写事务适合 v0.3.0 的单机流量；服务端高并发和多实例由 v0.4.0 PostgreSQL Store 与 v0.6.0 Redis 计数器承担。

进程启动时独占打开数据库；如果文件已经被其他进程锁定，启动失败并提示实例冲突，不以只读或无认证模式继续运行。

### 7.3 后续服务端实现

v0.4.0 增加 PostgreSQL Store；v0.6.0 使用 Redis 保存分布式限流、并发租约和短期 Credential 缓存。数据模型 ID 从 v0.3.0 起必须全局唯一，避免迁移时重新编号。

## 9. 网关 Principal

客户端认证不再只返回字符串 Key ID，而是返回：

```go
type Principal struct {
    TenantID      string
    ProjectID     string
    ClientID     string
    ClientName   string
    CredentialID string
    Policy       ClientPolicy
    Legacy       bool
}
```

网关调用记录新增：

- `client_id`
- `client_name_snapshot`
- `credential_id`
- `auth_source`：`managed`、`legacy`、`anonymous`
- `rejection_reason`

Credential ID 可用于内部诊断，但普通日志列表默认显示 Client 名称和脱敏前缀。

## 10. 请求鉴权与策略执行

### 10.1 Credential 提取

按顺序读取：

1. `Authorization: Bearer ...`
2. `x-api-key`
3. `x-goog-api-key`
4. Gemini 兼容的 `?key=`

如果多个位置同时存在不同 Key，返回 `400 invalid_request`，避免代理或客户端覆盖造成身份混淆。

公网模式默认禁止查询参数 Key；只有显式开启 Gemini 兼容时允许，并确保访问日志不记录该值。

### 10.2 前置检查

读取请求正文前完成：

- Credential 是否存在；
- Credential、Client 是否启用；
- 是否过期或撤销；
- 来源 IP 是否允许；
- 客户端协议是否允许；
- RPM 和突发限制的初步申请。

### 10.3 解码后检查

协议解码得到规范请求后执行：

- 客户端模型白名单；
- 单次 `max_output_tokens` 上限；
- 每日请求和 Token 预算；
- 每 Client 并发槽位；
- `/v1/models` 结果过滤；
- `/v1/messages/count_tokens` 使用相同权限。

权限判断必须发生在路由解析前，客户端不能通过 Route Header、Provider ID 或协议转换访问未授权目标。

### 10.4 响应错误

| HTTP | 错误码 | 场景 |
| --- | --- | --- |
| 401 | `authentication_error` | 缺少或无法识别 Key |
| 403 | `credential_disabled` | Credential 或 Client 被停用 |
| 403 | `credential_expired` | Credential 已过期 |
| 403 | `model_not_allowed` | 模型不在白名单 |
| 403 | `protocol_not_allowed` | 协议不允许 |
| 403 | `source_ip_not_allowed` | 来源 IP 不允许 |
| 429 | `rate_limited` | RPM 或突发限制 |
| 429 | `concurrency_limited` | Client 并发耗尽 |
| 429 | `quota_exhausted` | 日请求或 Token 预算耗尽 |

错误正文继续按照客户端协议转换，不能泄漏 Secret、数据库错误或其他 Client 信息。

## 11. 限流与用量

### 11.1 RPM

v0.3.0 使用单进程 Token Bucket：

- Key 为 Client ID，而非 Credential ID；
- 轮换 Credential 不会重置限流；
- 支持持续速率和 Burst；
- 进程重启后分钟级速率状态可以重置。

### 11.2 并发

每个 Client 使用独立信号量，另保留现有全局 `server.max_concurrent`。执行顺序为 Client 限制在前、全局限制在后，避免单个 Client 占满全部槽位。

### 11.3 每日配额

每日请求数必须在放行前原子增加。Token 配额分为：

- 请求前：根据本地 Token 估算和请求输出上限预留；
- 请求后：使用真实 Usage 多退少补；
- 无真实 Usage：使用现有本地估算并标记 `estimated`；
- 请求失败：释放尚未产生用量的预留；
- 上游可能已产生费用但未返回 Usage：记录独立诊断，不伪造精确成本。

v0.3.0 Token 配额是防止意外消耗的操作性限制，不宣称为财务级计费。

## 12. 控制台设计

新增侧边栏页面：`访问密钥`。界面采用 Key-first 交互；每次生成 Key 时，后端自动创建对应的 Client、Policy 和 Credential，用户无需理解内部 Client/Credential 层级。

### 12.1 列表

列包括：

- 客户端名称；
- Credential 前缀；
- 状态；
- 模型/协议范围；
- RPM/并发；
- 今日请求和 Token；
- 最近使用；
- 到期时间；
- 操作。

列表按 Credential 展开为一行一个 Key；轮换产生的新旧 Key 会分别显示，底层仍共享同一个 Client 与 Policy。

### 12.2 创建对话框

步骤：

1. 客户端名称；
2. 过期时间；
3. 模型和协议；
4. IP/CIDR；
5. RPM、Burst、并发和每日配额；
6. 选择要配置的应用；
7. 生成并配置；
8. 显示一次性 Secret 与应用验证结果。

默认策略：

- 本地回环监听：不过期、无限额，但仍建议每应用独立 Key；
- 非回环监听：默认 90 天过期，并要求显式确认无配额限制；
- 模型与协议默认允许当前全部路由；
- RPM、并发和每日配额默认关闭，避免升级后破坏现有调用。

### 12.3 应用配置页

访问密钥区域只显示一个输入框。用户可以直接输入外部 Key，也可以通过浏览器原生建议列表选择一个可恢复的托管 Key；输入框留空表示保留应用当前 Key。

页面不显示任何托管 Credential 的完整 Secret。选择托管 Key 时，输入框显示名称与安全前缀，浏览器只提交 Credential ID，服务端校验状态和所属 Client 后解密并写入；不可恢复或旧格式的 Key 不会进入建议列表。

## 13. 配置与兼容

新增启动配置建议：

```yaml
auth:
  enabled: true
  managed_store: true
  allow_query_key: false
  keys:
    - id: legacy-client
      value: ${AIROUTE_CLIENT_KEY}
```

兼容规则：

1. `managed_store` 缺省为 `true`，但没有 Client 时不自动启用认证。
2. `auth.keys` 继续加载为 Legacy Principal。
3. Managed Credential 优先按结构化 ID 查询，Legacy Key 只在不匹配新格式时遍历。
4. Legacy Key 可以在控制台迁移为标准 Credential；迁移保存摘要，不要求再次展示原值。
5. 删除最后一个有效 Credential 时必须提示网关将无法通过托管 Key 认证。
6. `auth.enabled: false` 时所有请求继续使用 `anonymous` Principal，但控制台明确显示“网关未启用客户端鉴权”。

## 14. 安全要求

1. Secret 不进入 `airoute.yaml`、普通配置 API、日志、指标、审计、错误或备份名称。
2. 一次性 Secret 页面关闭后不能恢复。
3. 前端不得写入 LocalStorage、SessionStorage 或 URL。
4. 管理 API 响应添加 `Cache-Control: no-store`。
5. 管理页面复制 Secret 后不自动写入剪贴板历史之外的存储。
6. 撤销立即影响新请求；进行中的流式请求不强制中断，审计记录其完成。
7. 系统时间回拨时，过期判断仍以当前安全时钟为准并记录异常。
8. 认证存储不可用时 fail closed；管理 API 返回服务不可用，不得回退匿名。
9. 查询参数 Key 必须在反向代理和应用访问日志层面同时脱敏。
10. 导出的诊断包只包含 Client ID、Credential ID、前缀和状态。
11. v0.3.0 管理 API 必须拒绝非回环来源；不能只依赖浏览器同源检查或 Host Header。
12. 状态库备份与 HMAC 主密钥必须成套校验和恢复，恢复失败时不得退回 Legacy 或匿名鉴权。

## 15. 迁移流程

### 15.1 从 v0.2.6 升级

首次启动：

1. 创建空状态库；
2. 保持现有 `auth.keys` 原样工作；
3. 控制台显示 Legacy Key 数量；
4. 用户可选择“迁移到客户端密钥”；
5. 系统为每个 Legacy Key 创建 Client、Policy 和标准 Credential 摘要；
6. 验证新 Credential 与旧值匹配；
7. 保存状态后再从 YAML 移除对应 Key；
8. 配置保存失败时继续使用原 YAML，不产生半迁移。

### 15.2 无中断启用鉴权

当前 `auth.enabled: false` 时：

1. 创建新 Client 和 Credential；
2. 将 Secret 配置到选中的应用；
3. 在仍允许匿名访问时执行带 Key 的验证；
4. 全部关键应用验证成功后才允许开启 `auth.enabled`；
5. 开启后再次验证；
6. 失败时自动恢复认证开关，但保留已创建 Credential。

## 19. 为公网路线预留的边界

v0.3.0 不实现租户管理，但 Client、Policy、Usage 和 Audit 从首次写入起就保存 `TenantID` 与 `ProjectID`；Credential 通过 Client 继承范围。Store 查询必须显式接收或校验所属范围，不得假设永久只有一个管理员。后续版本计划：

- v0.4.0：PostgreSQL、管理员登录、RBAC、审计、TLS/可信代理和公网安全模式；
- v0.5.0：Tenant、Project、Usage Plan、自助 Key 和持久化用量账本；
- v0.6.0：Redis 分布式限流、控制面/数据面/Worker 拆分和多实例；
- v0.7.0：版本化定价、余额或套餐、财务级预扣结算和对账。

v0.3.0 使用的 `tenant_default` 和 `project_default` 在升级后转为真实 Tenant 与 Project 记录，Client ID 和历史用量不改变。
